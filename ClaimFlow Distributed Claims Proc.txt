# ClaimFlow: Distributed Claims Processing Platform

## Overview

**ClaimFlow** is a modern distributed claims processing system that simulates enterprise-grade insurance claim handling. Built with FastAPI backend and React frontend, it demonstrates microservices architecture with real-time processing and monitoring.

### Key Features
- **Distributed Processing**: 5 parallel workers handle claims asynchronously
- **Real-time Dashboard**: Live metrics and claim status updates
- **Fault Tolerance**: Dead Letter Queue for failed claims
- **WebSocket Updates**: Instant UI updates without refresh
- **Production Ready**: Scalable architecture for enterprise deployment

---

## Architecture

### System Flow
```
User → React UI → FastAPI API → Input Queue → Workers → Output/DLQ
                    ↓
            WebSocket ← Real-time Updates
```

### Core Components

#### Backend (FastAPI + Python)
- **API Gateway**: REST endpoints for claim management
- **Queue Engine**: In-memory Service Bus (3 queues)
- **Consumer Workers**: 5 parallel async processors
- **Data Store**: In-memory persistence layer
- **WebSocket Manager**: Real-time broadcasting

#### Frontend (React + Vite)
- **Dashboard**: Real-time metrics and charts
- **Claims View**: Claim management interface
- **Logs View**: Audit trail viewer
- **WebSocket Client**: Live data synchronization

---

## Key Components

### 1. Claims Processing

#### Claim Lifecycle
1. **Submission**: User creates claim via UI/API
2. **Queueing**: Claim stored and pushed to input queue
3. **Processing**: Worker picks up claim, simulates processing (4-6s)
4. **Outcome**: 80% success → output queue, 20% failure → DLQ
5. **Updates**: Real-time status broadcasts to UI

#### Processing Simulation
- Random delay: 4-6 seconds
- Success rate: 80% (configurable)
- Error types: Validation, timeout, data integrity, etc.

### 2. Queue System

#### Queue Types
- **Input Queue**: Claims awaiting processing
- **Output Queue**: Successfully processed claims
- **Dead Letter Queue (DLQ)**: Failed claims for manual review

#### Queue Operations
- **Enqueue**: Add claims to input queue
- **Dequeue**: Workers pull claims for processing
- **Routing**: Success/failure determines destination queue

### 3. Real-time Communication

#### WebSocket Features
- **Claim Updates**: Live status changes
- **Metrics Updates**: Real-time performance data
- **Connection Management**: Auto-reconnect on failures

#### Message Types
```javascript
// Claim status update
{ type: "claim_update", data: { claim_id: "CLM-123", status: "Processing" } }

// System metrics
{ type: "metrics_update", data: { success_rate: 80.5, throughput: 0.023 } }
```

## Troubleshooting

### Common Issues

#### Backend Won't Start
```bash
# Check Python version
python --version

# Activate virtual environment
venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt
```

#### Frontend Build Fails
```bash
# Clear cache and reinstall
rm -rf node_modules package-lock.json
npm install
```

#### WebSocket Not Connecting
- Check backend is running on port 8000
- Verify CORS settings in config.py
- Check browser console for connection errors

#### Claims Stuck in Processing
- Worker may have crashed
- Check backend logs for errors
- Restart backend to reset workers

### Performance Issues

#### High Memory Usage
- Large number of claims in memory
- Consider external database for production
- Implement claim cleanup policies

#### Slow Processing
- Check worker count (default: 5)
- Monitor queue sizes
- Scale horizontally with more instances

#### WebSocket Lag
- Too many concurrent connections
- Network latency issues
- Implement connection limits

## 2. System Architecture

### 2.1 High-Level Architecture

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│   React UI      │◄────┤   FastAPI API     │◄────┤   Input Queue    │
│   (Dashboard,   │ ws  │   Gateway         │     │   (Service Bus)  │
│   Claims, Logs) │     └──────────────────┘     └─────────┬───────┘
└─────────────────┘                                         │
                                                  ┌────────▼────────┐
                                                  │  5 Consumer     │
                                                  │  Workers        │
                                                  │  (Parallel)     │
                                                  └────────┬────────┘
                                                           │
                                             ┌────────────┴────────────┐
                                             ▼                         ▼
                                  ┌─────────────────┐        ┌─────────────────┐
                                  │  Output Queue   │        │   Dead Letter    │
                                  │  (Success)      │        │   Queue (DLQ)    │
                                  └─────────────────┘        └─────────────────┘
```

### 2.2 Component Overview

#### Backend Components
- **API Gateway (FastAPI)**: RESTful API endpoints for claim management
- **Queue Engine**: In-memory Service Bus simulator with 3 queues
- **Consumer Workers**: 5 parallel async coroutines processing claims
- **Data Store**: In-memory persistence layer for claims and logs
- **WebSocket Manager**: Real-time broadcasting service
- **Configuration**: Centralized settings and constants

#### Frontend Components
- **Dashboard**: Real-time metrics and processing charts
- **Claims Management**: Claim submission, viewing, and retry functionality
- **Logs Viewer**: Comprehensive audit trail interface
- **WebSocket Client**: Real-time data synchronization

### 2.3 Data Flow

1. **Claim Submission**: UI/API → Database → Input Queue
2. **Processing**: Workers dequeue → Process → Success/Failure routing
3. **Updates**: WebSocket broadcasts → UI real-time updates
4. **Monitoring**: Metrics collection → Dashboard visualization

### 2.4 Technology Choices Rationale

- **FastAPI**: High-performance async framework with auto-generated OpenAPI docs
- **React**: Component-based UI with excellent ecosystem for real-time apps
- **WebSocket**: Bidirectional communication for real-time features
- **In-memory Storage**: Simplicity for demo while maintaining production patterns
- **Async/Await**: Non-blocking I/O for high concurrency

## 8. Queue Management

### 8.1 Queue Architecture

The system implements three primary queues using an in-memory Service Bus simulator:

#### 8.1.1 Input Queue
- **Purpose**: Claims awaiting processing
- **Source**: API claim creation
- **Consumer**: Worker processes
- **Naming**: `input_queue`

#### 8.1.2 Output Queue
- **Purpose**: Successfully processed claims
- **Source**: Worker completion (success)
- **Consumer**: External systems (future)
- **Naming**: `output_queue`

#### 8.1.3 Dead Letter Queue (DLQ)
- **Purpose**: Failed claims requiring manual review
- **Source**: Worker completion (failure)
- **Consumer**: Manual processing or retry
- **Naming**: `dead_letter_queue`

### 8.2 Queue Operations

#### 8.2.1 Enqueue (Send)
```python
await service_bus.send(queue_name, message_dict)
```

**Features**:
- Async operation (non-blocking)
- Automatic metrics tracking
- Thread-safe implementation

#### 8.2.2 Dequeue (Receive)
```python
message = await service_bus.receive(queue_name, timeout=2.0)
```

**Features**:
- Configurable timeout
- FIFO (First In, First Out) ordering
- Returns `None` if queue empty

### 8.3 Queue Monitoring

#### 8.3.1 Size Tracking
```python
input_size = service_bus.input_queue_size
output_size = service_bus.output_queue_size
dlq_size = service_bus.dlq_size
```

#### 8.3.2 Health Metrics
- **Queue Depth**: Current number of messages
- **Processing Rate**: Messages processed per second
- **Error Rate**: Failed messages as percentage

### 8.4 Message Format

All queue messages follow a consistent structure:
```python
{
    "claim_id": "CLM-A1B2C3D4",
    "claimant_name": "John Doe",
    "policy_number": "POL-001",
    "loss_date": "2026-01-15",
    "loss_description": "Vehicle collision",
    "amount": 5000.00,
    "created_at": "2026-01-15T10:30:00Z",
    "status": "Completed",  # Added during processing
    "error": "Error message"  # Only for DLQ messages
}
```

### 8.5 Scalability Considerations

#### 8.5.1 Current Implementation
- **In-memory**: Fast but not persistent
- **Single Instance**: No distribution
- **Limited Capacity**: Memory constraints

#### 8.5.2 Production Migration
- **Azure Service Bus**: Cloud-native message broker
- **RabbitMQ**: Open-source message queue
- **Redis**: In-memory data structure store
- **Kafka**: High-throughput distributed streaming

### 8.6 Error Handling

#### 8.6.1 Message Poisoning
- Automatic DLQ routing for failed messages
- Prevents infinite retry loops
- Manual intervention required for DLQ messages

#### 8.6.2 Queue Full Conditions
- Current implementation: Unlimited growth
- Production: Implement backpressure
- Monitoring: Alert on queue depth thresholds

## 10. Processing Workflow

### 10.1 Claim Lifecycle

#### 10.1.1 Phase 1: Submission
1. **User Input**: Claim details entered via UI
2. **Validation**: Frontend and backend validation
3. **Persistence**: Claim stored in database
4. **Queueing**: Claim pushed to input queue
5. **Acknowledgment**: Success response to user

#### 10.1.2 Phase 2: Processing
1. **Dequeue**: Worker picks claim from input queue
2. **Status Update**: Claim marked as "Processing"
3. **Broadcast**: Real-time status update sent
4. **Simulation**: Configurable processing delay (4-6 seconds)
5. **Outcome**: Success (80%) or Failure (20%)

#### 10.1.3 Phase 3: Completion
**Success Path**:
1. **Status Update**: Claim marked "Completed"
2. **Queue Routing**: Moved to output queue
3. **Logging**: Success log entry created
4. **Broadcast**: Final status update

**Failure Path**:
1. **Status Update**: Claim marked "Error"
2. **Error Details**: Failure reason recorded
3. **DLQ Routing**: Moved to dead letter queue
4. **Logging**: Error log entry created
5. **Broadcast**: Error status update

### 10.2 Worker Architecture

#### 10.2.1 Worker Initialization
```python
# 5 workers started on application startup
for i in range(1, NUM_CONSUMERS + 1):
    task = asyncio.create_task(consumer_worker(i))
    _worker_tasks.append(task)
```

#### 10.2.2 Worker Loop
```python
async def consumer_worker(worker_id: int):
    while True:  # Continuous processing
        message = await service_bus.receive(INPUT_QUEUE, timeout=2.0)
        if message is None:
            continue  # No work available
        
        # Process claim...
```

#### 10.2.3 Processing Logic
1. **Claim Retrieval**: Load full claim data
2. **Status Transition**: PENDING → PROCESSING
3. **Processing Simulation**: Random delay with failure probability
4. **Outcome Determination**: Success/Failure based on random chance
5. **Result Handling**: Update claim, route to appropriate queue
6. **Cleanup**: Broadcast updates, log completion

### 10.3 Error Scenarios

#### 10.3.1 Processing Failures
- **Validation Errors**: Missing or invalid data
- **External Service Failures**: Simulated API timeouts
- **Data Integrity Issues**: Corrupted claim data
- **Coverage Verification**: Policy validation failures
- **Payment Gateway Issues**: Settlement processing errors

#### 10.3.2 System Failures
- **Worker Crashes**: Individual worker failure
- **Queue Full**: Input queue capacity exceeded
- **Database Issues**: Persistence layer failures
- **Network Problems**: Inter-service communication failures

### 10.4 Retry Mechanisms

#### 10.4.1 Automatic Retry
- **Not Implemented**: Current system uses DLQ for failures
- **Future Enhancement**: Configurable retry policies
- **Exponential Backoff**: Progressive delay between attempts

#### 10.4.2 Manual Retry
```http
POST /claims/retry/{claim_id}
```
- **Status Reset**: ERROR → PENDING
- **Re-queueing**: Claim returned to input queue
- **Retry Tracking**: Increment retry counter

### 10.5 Performance Characteristics

#### 10.5.1 Throughput
- **Target**: 5 claims/minute (with 5-second average processing)
- **Peak Capacity**: 12 claims/minute (with parallel workers)
- **Scalability**: Linear scaling with additional workers

#### 10.5.2 Latency
- **Average Processing Time**: 4.75 seconds
- **P95 Latency**: ~6 seconds (max delay)
- **Queue Wait Time**: Variable (depends on load)

#### 10.5.3 Reliability
- **Success Rate**: 80% (configurable)
- **Error Rate**: 20% (configurable)
- **MTTR**: Manual intervention required for DLQ items

- **Maximum Retries**: Prevent infinite retry loops
- **Dead Letter Queues**: Failed messages routed for manual review

#### 14.2.3 Error Propagation
- **HTTP Status Codes**: Appropriate status codes for API errors
- **Error Messages**: Clear, actionable error descriptions
- **Stack Traces**: Detailed error information for debugging

### 14.3 Error Recovery

- **Worker Restart**: Failed workers automatically restarted
- **Connection Recovery**: Automatic reconnection for network issues
- **State Synchronization**: Recover from inconsistent states

#### 14.3.2 Manual Recovery

## 19. Troubleshooting Guide

### 19.1 Common Issues

#### 19.1.1 Backend Startup Issues
**Problem**: Backend fails to start
```
uvicorn app.main:app --reload
# Error: ModuleNotFoundError: No module named 'fastapi'
```

**Solution**:
```bash
# Ensure virtual environment is activated
source venv/bin/activate  # Linux/Mac
# or
venv\Scripts\activate     # Windows

# Install dependencies
pip install -r requirements.txt
```

#### 19.1.2 Frontend Build Issues
**Problem**: Frontend build fails
```
npm run dev
# Error: Cannot resolve dependency
```

**Solution**:
```bash
# Clear node_modules and reinstall
rm -rf node_modules package-lock.json
npm install
```

#### 19.1.3 WebSocket Connection Issues
**Problem**: Real-time updates not working
- Check browser developer tools for WebSocket errors
- Verify backend is running on correct port
- Check CORS configuration

**Solution**:
```javascript
// Test WebSocket connection
const ws = new WebSocket('ws://localhost:8000/ws');
ws.onopen = () => console.log('Connected');
ws.onerror = (error) => console.error('WebSocket error:', error);
```

### 19.2 Performance Issues

#### 19.2.1 High Memory Usage
**Symptoms**: Application consuming excessive memory
**Causes**: Large number of claims in memory, memory leaks

**Solutions**:
- Implement claim cleanup policies
- Add memory monitoring and alerts
- Consider external database for persistence

#### 19.2.2 Slow API Responses
**Symptoms**: API endpoints responding slowly
**Causes**: High load, inefficient queries, resource contention

**Solutions**:
- Add database indexes
- Implement caching
- Scale horizontally with more instances

#### 19.2.3 Queue Backlog
**Symptoms**: Input queue growing rapidly
**Causes**: Workers unable to keep up with claim volume

**Solutions**:
- Increase number of worker processes
- Optimize processing logic
- Implement load shedding

### 19.3 Data Issues

#### 19.3.1 Claim Status Stuck
**Problem**: Claims stuck in "Processing" status
**Cause**: Worker crashed during processing

**Solution**:
```python
# Manual status reset (development only)
await store.update_claim(claim_id, status=ClaimStatus.PENDING)
```

#### 19.3.2 Missing Metrics
**Problem**: Dashboard showing no metrics
**Cause**: Metrics calculation failing

**Solution**:
- Check application logs for errors
- Verify queue sizes are being tracked
- Restart application to reset metrics

#### 19.3.3 WebSocket Disconnects
**Problem**: Real-time updates stop working
**Cause**: Network issues or server restart

**Solution**:
- Implement automatic reconnection in frontend
- Add connection health monitoring
- Check server logs for WebSocket errors

### 19.4 Development Issues

#### 19.4.1 Import Errors
**Problem**: Python import errors during development
**Cause**: Module not installed or path issues

**Solution**:
```bash
# Install in development mode
pip install -e .

# Check Python path
python -c "import sys; print(sys.path)"
```

#### 19.4.2 Hot Reload Not Working
**Problem**: Code changes not reflected immediately
**Cause**: File watching issues or cache problems

**Solution**:
```bash
# Clear Python cache
find . -name "*.pyc" -delete
find . -name "__pycache__" -type d -exec rm -rf {} +

# Restart development server
```

#### 19.4.3 CORS Errors
**Problem**: Frontend unable to connect to backend
```
Access to XMLHttpRequest at 'http://localhost:8000' 
from origin 'http://localhost:5173' has been blocked by CORS policy
```

**Solution**:
- Add frontend origin to CORS_ORIGINS in config.py
- Restart backend server
- Check network firewall settings

### 19.5 Production Issues

#### 19.5.1 Database Connection Issues
**Problem**: Unable to connect to production database
**Cause**: Network issues, credential problems, or service downtime

**Solution**:
- Verify connection string and credentials
- Check network connectivity
- Implement connection pooling and retry logic

#### 19.5.2 Certificate Issues
**Problem**: SSL/TLS certificate problems
**Cause**: Expired certificates or misconfiguration

**Solution**:
- Renew certificates before expiration
- Verify certificate chain
- Check certificate store configuration

#### 19.5.3 Load Balancing Issues
**Problem**: Requests not distributing properly
**Cause**: Load balancer configuration or health check failures

**Solution**:
- Verify health check endpoints
- Check load balancer logs
- Adjust load balancing algorithm
